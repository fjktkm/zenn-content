## 1. はじめに

### 1.1. クリエイティブソフトウェアの問題点

グラフィックデザインの制作には Adobe Illustrator や Affinity などのクリエイティブソフトウェアを使うのが一般的です．しかし，このような方法には共同編集が難しいという問題があります．たとえば，次のような要素によってしばしば作業環境の再現性を保つことが難しいです．

- ユーザー側の設定のカスタマイズ
- 素材やフォントなどの外部依存関係の管理

また，PNG 複数サイズ・SVG・PDF など複数フォーマットのアセット書き出しも手作業に頼る必要があり，その点でも一貫性のある運用を実現することが難しいです．

### 1.2. Creative x DevOps

一方，ソフトウェア開発では DevOps の手法が普及しています．Git でのバージョン管理・GitHub による共同編集・CI/CD による自動デプロイといったワークフローが当たり前になっています．こうした手法をクリエイティブのワークフローにも持ち込むことができれば，そうしたクリエイティブの運用上の課題を解決できるのではないかと考えられます．

しかし，クリエイティブソフトウェアに DevOps を適用するにもいろいろと障壁があります．たとえば，クリエイティブソフトウェアのプロジェクトファイルは基本的にバイナリ形式で保存されているため，差分の確認やマージができず，通常の Git ワークフローが利用できません．また，アセットの書き出しも GUI ツールを介する以上は自動化しにくく，CI/CD に組み込むことが困難です．

こうした問題を解決するアプローチとして，もうクリエイティブソフトウェアを利用するのもやめてしまって，ソースコードでグラフィックデザインを定義するという方法があります．形状・色・質感をコードで記述することで Git でのバージョン管理が可能になり，アセットはコマンド一発で生成できます．クリエイティブソフトウェアを使わないので CI/CD への組み込みも容易で，GitHub Pages などを使って自動的に公開することもできます．

本記事では，Rust と Cairo を用いてロゴアセットをソースコードベースで管理するリポジトリを作った話を紹介します．

https://github.com/fjktkm/logo

## 2. 方法

### 2.1. 生成するアセット

生成するアセットは以下のバリエーションで出力します：

- **フォーマット**: PNG・SVG・PDF
- **テーマ**: ライトテーマ / ダークテーマ
- **サイズ（PNG のみ）**: 256px・512px・1024px

実際に生成されるロゴのデザインは以下のとおりです．

| ライトテーマ                                                 | ダークテーマ                                                |
| ------------------------------------------------------------ | ----------------------------------------------------------- |
| ![ライトテーマのロゴ](/images/2a29874805d0b8/logo_light.png) | ![ダークテーマのロゴ](/images/2a29874805d0b8/logo_dark.png) |

### 2.2. 技術スタック

| 役割           | 選択           |
| -------------- | -------------- |
| 言語           | Rust           |
| 描画ライブラリ | cairo-rs       |
| CI/CD          | GitHub Actions |
| ホスティング   | GitHub Pages   |

描画ライブラリとして [cairo-rs](https://gtk-rs.org/gtk-rs-core/stable/latest/docs/cairo/) を選びました．Cairo は SVG・PDF・PNG の 3 フォーマットすべてを 1 つの API で扱えます．描画ライブラリにはいろいろあるのですが，出力フォーマットの要件を考えると Cairo が最適だと判断しました．

なお，Rust を選んだのは個人の好みというだけで，別にほかの言語のバインディングでも同様のことは可能だと思います．

## 3. 実装

コードの構成は以下のとおりです：

```
src/
├── lib.rs      # 公開 API
├── main.rs     # エントリーポイント
├── shape.rs    # ロゴのシェイプ定義
├── theme.rs    # テーマ・色の定義
├── noise.rs    # ガウスノイズの生成
└── render.rs   # 描画処理
```

### 3.1. シェイプ（shape.rs）

ロゴの形状はベジェ曲線で定義します．Cairo の `move_to` / `curve_to` を使って，曲線からなるパスを組み立てます．SVG を直書きしているような感じですね．

```rust
pub fn build_path(cr: &Context, width: f64, height: f64) {
    cr.save().unwrap();
    cr.scale(width / SHAPE_WIDTH, height / SHAPE_HEIGHT);

    cr.move_to(21.0, 0.0);
    cr.curve_to(42.0, 0.0, 42.0, 0.0, 63.0, 21.0);
    cr.curve_to(84.0, 42.0, 84.0, 42.0, 84.0, 75.0);
    // ...
    cr.close_path();

    cr.restore().unwrap();
}
```

`scale()` で正規化することで，同じ関数を PNG の任意解像度と SVG・PDF の描画にそのまま使い回せるようにしています．シェイプを変更したければここだけ書き換えればよく，変更履歴は Git で追跡できます．

### 3.2. テーマ（theme.rs）

テーマは `Theme` 構造体で表現します．

```rust
pub struct Theme {
    pub name: &'static str,
    pub background: Color,
    pub gradient_start: Color,
    pub gradient_end: Color,
    pub bg_noise_intensity: f64,
    pub shape_noise_intensity: f64,
    pub noise_seed: u64,
}
```

シェイプ内部はグラデーションで塗ります．ライトテーマは背景が白で背景ノイズなし，ダークテーマは暗い背景色に背景ノイズを乗せる設定にしています．こうした少し込み入った設定についても，テーマのパラメータとして持たせることでソースコードで明示的に管理できるようにしています．

### 3.3. ノイズ（noise.rs）

画像にノイズのテクスチャを与えるため，ガウスノイズを各ピクセルに加算しています．

もともと Affinity でロゴを作っていたため，Affinity でのノイズエフェクトと同様の見た目になるようにガウスノイズを実装しました．Affinity のノイズエフェクトのパネルには「強度」というパラメータがあり，これによってノイズの強さが変わります．

この値が実際に何を意味しているのかはドキュメントで明示されていないためいろいろと試行錯誤して検証したのですが，どうやら 6σ の幅を「強度」としているらしいということがわかりました．言われてみれば妥当な定義ですね．この定義に基づいて，ノイズを付加するように実装しました．

### 3.4. レンダリング（render.rs）

描画の流れは以下のとおりです：

1. 背景色で塗りつぶした `ImageSurface` を作成し，背景ノイズを加算する
2. グラデーションを描いた `ImageSurface` を作成し，シェイプノイズを加算する
3. 出力サーフェスに背景を転写し，グラデーション済みシェイプをクリッピングして合成する

PNG の場合は `ImageSurface` に直接描いて PNG ファイルとして書き出します．SVG・PDF の場合は `SvgSurface` / `PdfSurface` をターゲットとして同じ描画処理を実行します．

```rust
pub fn render_svg(path: &Path, size: f64, theme: &Theme) -> Result<()> {
    let surface = SvgSurface::new(size, size, Some(path))?;
    let cr = Context::new(&surface)?;
    draw_logo(&cr, size, theme)?;
    drop(cr);
    surface.finish();
    Ok(())
}
```

まとめてノイズを加算せずに，背景とシェイプそれぞれにノイズを加算しているのは，ベクター形式で書き出した際にアウトラインがラスタライズされてしまうのを防ぐためです．ノイズを加算してからクリッピングすることで，アウトラインがベクター形式のまま保持されるようにしています．

### 3.5. SVG のベクター品質テスト

SVG 出力において，ロゴのアウトラインがベクターのパス要素として保持されているかどうかをテストで確認しています．エフェクトを掛ける順番によっては，ベクターで出力できるはずの要素が意図せずラスタライズされてしまうことがあります．このテストによって，そうした問題が起きていないことを明示的に保証しています．

```rust
#[test]
fn svg_has_vector_clip_path() {
    let dir = tempfile::tempdir().expect("tempdir");
    let path = dir.path().join("test.svg");
    render_svg(&path, 192.0, &DARK).expect("render_svg");
    let content = std::fs::read_to_string(&path).expect("read svg");
    assert!(content.contains("<clipPath"), "SVG contains no <clipPath> element");
    // clipPath 内にベジェ曲線を含む <path> があることを確認する
    // ...
}
```

`<clipPath>` 要素内にベジェ曲線（`C` コマンド）を含む `<path>` が存在することを確認することで，クリッピング経路がラスター化されずパス要素として保持されていることを担保しています．SVG 全体が純ベクターであることを保証するものではありませんが，アウトラインの品質を守る回帰テストとして機能しています．

特にコーディングエージェントを使っていると，意図せずこうした変更が実行されてしまうことがあります．Vision Model は基本ビットマップ形式しか対応していないので，そりゃあなたにとっては同じに見えるんでしょうけどね～という感じです．そのため，こうしたアウトラインに関するテストを明示的に含めておくことには重要な意味があるのではないかと思います．

### 3.6. CI/CD

次のような GitHub Actions のワークフローにより，フォーマット・Lint・テスト・アセット生成・デプロイを自動化しています．

```yaml
steps:
  - name: Install Cairo
    run: sudo apt-get update && sudo apt-get install -y libcairo2-dev
  - name: Check formatting
    run: cargo fmt --check
  - name: Run linter
    run: cargo clippy --all-targets --all-features -- -D warnings
  - name: Run tests
    run: cargo test
  - name: Generate assets
    run: cargo run --release
  - name: Upload Pages artifact
    uses: actions/upload-pages-artifact@v4
    with:
      path: dist/
```

`main` ブランチへの Push をトリガーに，アセットを生成して GitHub Pages へデプロイします．また，`gh release create` で GitHub Release も自動作成するようにしており，生成されたアセットをリリースページからも直接ダウンロードできます．ロゴをちょっと修正したくなったら，コードを変更して Push するだけで最新のアセットが自動的に公開されるというわけです．

## 4. 考察

この手法が有効なのは，**幾何学的に定義できる** デザインに限られます．ロゴやアイコンのような比較的シンプルなデザイン，ブランドアセットやプレスキットのように複数のフォーマット・サイズでの出力が必要なケースなど，特定のユースケースにおいては非常に有効なアプローチだと感じています．一方で，複雑なイラストや写真ベースのデザインをコードで表現しようとすると，かえって作業が困難になるでしょう．

また，デザインの試行錯誤は GUI ツールのほうが格段に速いという現実があります．「どんなシェイプにするか」を決めるまでの探索段階では，GUI での検討が有効です．コードによる管理が真価を発揮するのは，形を決めた後の段階です．

## 5. おわりに

Rust + Cairo を使うことで，ロゴの形状・色・質感をすべてコードで定義し，複数のフォーマットと解像度のアセットをまとめて生成できる仕組みを作りました．また，GitHub Actions を使って CI/CD を構築し，Push するだけで最新のアセットが自動的に公開されるようにしました．ロゴのような比較的シンプルなデザインにおいては，この手法はなかなかよく機能すると感じています．

結局のところ，Adobe や Affinity がこうしたワークフローに対応してくれるというのが一番好ましいことだろうと思います．そうすれば，クリエイティブソフトウェアの利便性を享受しつつ，Git でのバージョン管理や CI/CD も自然に利用できるようになるわけですから．このリポジトリはある種の概念実証のようなもので，そうした未来に向けた一つの可能性を示すものであればよいなと思います．
