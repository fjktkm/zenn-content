## 1. はじめに

この記事では Dev Container を用いて，Python のパッケージマネージャーである uv の環境を構築する方法を解説します．できるだけオフィシャルなツールだけに依存することを重要視し，uv の公式イメージと Dev Container の機能だけを使って構成する手法を提案します．

本記事で紹介する構成は以下のリポジトリで公開しています．

https://github.com/fjktkm/uv-devcontainer

## 2. 背景

### 2.1. Python テンプレートは pip 前提

Dev Container の公式テンプレートの中には Python 用のものが存在します．

https://github.com/devcontainers/templates/tree/main/src/python

しかしこのテンプレートは，Python のバージョンごとに用意された `mcr.microsoft.com/devcontainers/python` イメージから使いたいバージョンを選ぶ形式になっており，pip や venv を使う従来の Python 開発を前提とした構成です．uv はパッケージ管理だけでなく Python 本体のインストールやバージョン管理まで担うツールなので，あらかじめ特定バージョンの Python が焼き込まれたイメージを選ぶという発想とは相性がよくありません．

### 2.2. uv Feature の採用は避けたい

公式テンプレートが使えないとなると，次に候補になるのは Features です．Features は，ベースイメージに開発用のツールを追加するための仕組みです．

https://containers.dev/features

uv をインストールする Feature を探してみると，`ghcr.io/jsburckhardt/devcontainer-features/uv`，`ghcr.io/va-h/devcontainers-features/uv`，`ghcr.io/gvatsal60/dev-container-features/uv` など，複数の Feature が見つかります．

ただし，これらはいずれも Dev Container の公式 Feature ではなく，uv を開発する Astral が提供しているものでもありません．このような Feature に依存することは信頼性や保守性の観点から好ましくありません．

また，Feature に向いているのは，Git などの開発用の補助ツールです．uv のような実行環境の中心になるツールまで Feature に任せると，ベースイメージと Feature の役割分担が曖昧になります．uv はベースイメージ側で用意したいところです．

### 2.3. uv には公式イメージがある

幸い，uv は公式の Docker イメージを提供しています．これをベースイメージにすれば，Feature に頼らずに uv 本体を用意できます．

https://docs.astral.sh/uv/guides/integration/docker/

一方で，`mcr.microsoft.com/devcontainers/python` のような Dev Container 用のイメージは，uv 公式にはまだ存在しません．uv のリポジトリにはこれを求める Issue が立てられていますが，本記事の執筆時点でも Open のままで，優先度も高くない状態です．そのため，公式イメージをどう Dev Container に組み込むかは自分で考える必要があります．

https://github.com/astral-sh/uv/issues/8737

## 3. 提案手法

本記事では uv の公式イメージをベースイメージとして採用し，`devcontainer.json` の機能だけで uv の実行環境を構築します．

### 3.1. ベースイメージ

まず `.devcontainer/devcontainer.json` を次のように作成します．

```jsonc:.devcontainer/devcontainer.json
{
  "name": "Python (uv)",
  "image": "ghcr.io/astral-sh/uv:debian"
}
```

`debian` タグは buildpack-deps ベースの Debian イメージに uv バイナリだけを追加したもので，Python 本体は含まれていません．Python のインストールとバージョン管理は，ベースイメージではなく uv 自身に任せる方針です．

### 3.2. common-utils Feature の追加

次に，`common-utils` Feature を追加します．これにより，Git などの基本的な開発ツール，シェル設定，非 root ユーザーが追加され，Microsoft 謹製イメージに近い使い勝手になります．

```diff jsonc:.devcontainer/devcontainer.json
 {
   "name": "Python (uv)",
-  "image": "ghcr.io/astral-sh/uv:debian"
+  "image": "ghcr.io/astral-sh/uv:debian",
+  "features": {
+    "ghcr.io/devcontainers/features/common-utils:2": {}
+  },
+  "remoteUser": "vscode"
 }
```

### 3.3. 依存関係の同期

続いて，プロジェクトの依存関係を同期する処理を追加します．Dev Container では，依存関係の同期は `postCreateCommand` で行うのが一般的です．これは，依存関係は開発中に変更される可能性があるため，イメージのビルド時ではなく，ビルドしたコンテナ内で行うのが望ましいからです．

```diff jsonc:.devcontainer/devcontainer.json
 {
   "name": "Python (uv)",
   "image": "ghcr.io/astral-sh/uv:debian",
   "features": {
     "ghcr.io/devcontainers/features/common-utils:2": {}
   },
-  "remoteUser": "vscode"
+  "postCreateCommand": "uv sync",
+  "remoteUser": "vscode"
 }
```

### 3.4. 拡張機能の設定

最後に VS Code の拡張機能を設定します．

```diff jsonc:.devcontainer/devcontainer.json
 {
   "name": "Python (uv)",
   "image": "ghcr.io/astral-sh/uv:debian",
   "features": {
     "ghcr.io/devcontainers/features/common-utils:2": {}
   },
   "postCreateCommand": "uv sync",
-  "remoteUser": "vscode"
+  "customizations": {
+    "vscode": {
+      "extensions": [
+        "ms-python.python",
+        "tamasfe.even-better-toml"
+      ]
+    }
+  },
+  "remoteUser": "vscode"
 }
```

`ms-python.python` は Python の公式拡張機能です．`tamasfe.even-better-toml` は `pyproject.toml` や `uv.lock` などの TOML ファイルの構文ハイライトと補完を提供します．

完成した `devcontainer.json` は次のようになります．

```jsonc:.devcontainer/devcontainer.json
{
  "name": "Python (uv)",
  "image": "ghcr.io/astral-sh/uv:debian",
  "features": {
    "ghcr.io/devcontainers/features/common-utils:2": {}
  },
  "postCreateCommand": "uv sync",
  "customizations": {
    "vscode": {
      "extensions": [
        "ms-python.python",
        "tamasfe.even-better-toml"
      ]
    }
  },
  "remoteUser": "vscode"
}
```

### 3.5. キャッシュと仮想環境のボリューム化

ここまでの構成でも uv を使った開発は問題なく行えますが，さらに次のような最適化を加えることもできます．

```diff jsonc:.devcontainer/devcontainer.json
 {
   "name": "Python (uv)",
   "image": "ghcr.io/astral-sh/uv:debian",
   "features": {
     "ghcr.io/devcontainers/features/common-utils:2": {}
   },
+  "mounts": [
+    {
+      "source": "uv",
+      "target": "/home/vscode/.cache/uv",
+      "type": "volume"
+    },
+    {
+      "target": "${containerWorkspaceFolder}/.venv",
+      "type": "volume"
+    }
+  ],
+  "containerEnv": {
+    "UV_LINK_MODE": "copy"
+  },
+  "onCreateCommand": "sudo chown -R vscode:vscode /home/vscode/.cache/uv .venv",
   "postCreateCommand": "uv sync",
   "customizations": {
     "vscode": {
       "extensions": [
         "ms-python.python",
         "tamasfe.even-better-toml"
       ]
     }
-  }
+  },
+  "remoteUser": "vscode"
 }
```

`.venv` には名前を指定しない匿名ボリュームをマウントしています．`.venv` は大量の小さいファイルで構成されるため，Windows や macOS ではホストとバインドマウントしたまま扱うと I/O が非常に遅くなることが知られています．ホストと同期する必要のない `.venv` をボリュームに切り出すことで，ホスト側を汚さず I/O も高速に保てます．このような設定はボリュームトリックと呼ばれ，Node.js の `node_modules` などでもよく使われます．uv の公式ドキュメントでも紹介されている構成です．

uv のキャッシュディレクトリには `uv` という名前付きボリュームをマウントしています．コンテナを作り直してもキャッシュが失われないため，`uv sync` の高速化を維持できます．PyTorch のような大きなライブラリを扱う場合は，このような手法が特に有効です．

ただし，キャッシュと `.venv` が別々のボリュームになると，uv がデフォルトで使うハードリンクを両者の間で貼れず，警告が出ます．そこで `UV_LINK_MODE` を `"copy"` に設定し，コピーで動作することを明示しています．

さらに，Docker の名前付きボリュームや匿名ボリュームは初回作成時に root 所有で作られるため，そのままでは非 root ユーザーである `vscode` から書き込めません．そこで `onCreateCommand` で所有者を `vscode` に変更しています．

## 4. おわりに

Dev Container による uv の環境構築では，公式の Python テンプレートが pip を前提にしている，uv の Feature はコミュニティ版しかない，uv 公式の Dev Container 用イメージも存在しない，といった悩ましさがあります．

一方で，uv の公式イメージをベースイメージにすれば，`devcontainer.json` の設定だけで uv の開発環境を構築できます．Python 本体のインストールやバージョン管理も uv 自身に任せられるため，オフィシャルなツールだけに依存したシンプルな構成に保てます．
