## 1. はじめに

この記事では Dev Container を用いて Haskell 環境を構築する方法を解説します．Dockerfile を書かず，Haskell の公式イメージとライフサイクルコマンドだけで構築する方針を採用します．

本記事で紹介する構成は以下のリポジトリで公開しています．

https://github.com/fjktkm/haskell-devcontainer

## 2. 背景

### 2.1. Dev Container に Haskell テンプレートがない

Dev Container で開発環境を構築しようとすると，まず公式テンプレートを探すことになると思います．Dev Container の公式テンプレートは `devcontainers/templates` リポジトリで管理されています．

https://github.com/devcontainers/templates

しかし，このリポジトリには Haskell のテンプレートがありません．以前は `microsoft/vscode-dev-containers` に Haskell 用の定義がありましたが，このリポジトリ自体はアーカイブされています．

https://github.com/microsoft/vscode-dev-containers/tree/main/containers/haskell

Dev Container の公式テンプレートで一発構築，という選択肢がないため，Haskell では別の構成を考える必要があります．

### 2.2. Haskell Feature の採用は避けたい

Dev Container にテンプレートがない場合，次に候補になるのは Features です．幸い，Haskell の Feature は存在します．

https://containers.dev/features#:~:text=ghcr.io/devcontainers%2Dextra/features/haskell

https://github.com/devcontainers-extra/features/tree/main/src/haskell

この Feature を使えば，GHC，Cabal，Stack，HLS などをまとめてインストールできます．VS Code の Haskell 拡張機能も動作します．

ただし，これで問題がすべて解決するわけではありません．この Feature は Dev Container Spec Maintainers が管理する公式 Feature ではなく，Haskell 公式のプロジェクトが提供しているものでもありません．完全に放置されているわけではないものの，Haskell Feature 単体で見ると活発にメンテナンスが行われているとは言いにくい状況です．

また，Feature は基本的に，ベースイメージに不足している開発用の補助ツールを追加するための仕組みです．Haskell 本体のような開発環境の中心になるツールチェーンまで Feature に任せると，ベースイメージと Feature の役割分担が曖昧になり，GHC，Cabal，Stack，HLS などのバージョンやインストール方法をどこで管理しているのかが見えにくくなります．言語処理系そのものは，ベースイメージに含めるほうが好ましいでしょう．

### 2.3. Haskell 公式イメージには HLS がない

Docker Hub には Haskell の公式イメージが公開されています．

https://hub.docker.com/_/haskell

このイメージには GHC，Cabal，Stack が含まれています．そのため，Haskell 本体の実行環境としてはこちらを使うのが自然です．

一方で，VS Code の Haskell 拡張機能のコア機能を担う Haskell Language Server（HLS）は含まれていません．そのため，公式イメージをそのまま Dev Container のベースイメージにすると，Haskell 拡張機能はそのままでは期待どおりに動作しません．

https://marketplace.visualstudio.com/items?itemName=haskell.haskell

HLS を公式イメージに含めることについては，過去に `haskell/docker-haskell` でも議論されています．そこでは，HLS は IDE やエディタと連携するためのツールであり，HLS のリリースや対応する GHC バージョンに強く依存するため，Haskell のコアとなる公式イメージに含めるよりも，別のイメージや Dev Container 側で扱うほうが自然ではないか，という意見が示されています．

https://github.com/haskell/docker-haskell/issues/41

https://github.com/haskell/docker-haskell/issues/76

### 2.4. Dockerfile を書くのは避けたい

Dockerfile を書けば自由にカスタマイズできますが，Dev Container ではなるべく避けたいです．公式イメージと Dev Container の標準機能で足りるのであれば，独自の Dockerfile は追加しないに越したことはありません．Dockerfile によるカスタマイズを増やすほど，どこで何をインストールしているのかが見えにくくなり，構成の保守性も下がります．

## 3. 提案手法

本記事では Haskell 公式イメージをベースイメージとして採用し，不足している HLS をライフサイクルコマンドでインストールします．Haskell Feature には依存せず，Dockerfile も書きません．

また，ライフサイクルコマンドには `postCreateCommand` ではなく `onCreateCommand` を使います．`onCreateCommand` はコンテナの初回作成時に実行されるコマンドです．HLS のインストールはコンテナの作成処理の一部と考えるのが自然なので，コンテナ作成後の処理である `postCreateCommand` よりも `onCreateCommand` のほうが意図に合っています．

### 3.1. ベースイメージ

まず `.devcontainer/devcontainer.json` を次のように作成します．

```jsonc:.devcontainer/devcontainer.json
{
  "name": "Haskell",
  "image": "haskell:latest"
}
```

### 3.2. common-utils Feature の追加

次に，`common-utils` Feature を追加します．これにより，Git などの基本的な開発ツール，シェル設定，非 root ユーザーが追加され，Microsoft 謹製イメージに近い使い勝手になります．

```diff jsonc:.devcontainer/devcontainer.json
 {
   "name": "Haskell",
-  "image": "haskell:latest"
+  "image": "haskell:latest",
+  "features": {
+    "ghcr.io/devcontainers/features/common-utils:2": {}
+  },
+  "remoteUser": "vscode"
 }
```

### 3.3. GHCup のインストール

HLS のインストールには GHCup を利用します．Haskell 公式イメージには GHCup が含まれていないため，まずは GHCup をインストールする設定を追加する必要があります．ここではライフサイクルコマンドと環境変数を活用して GHCup のインストールを行います．

```diff jsonc:.devcontainer/devcontainer.json
 {
   "name": "Haskell",
   "image": "haskell:latest",
   "features": {
     "ghcr.io/devcontainers/features/common-utils:2": {}
   },
+  "containerEnv": {
+    "BOOTSTRAP_HASKELL_NONINTERACTIVE": "1",
+    "BOOTSTRAP_HASKELL_MINIMAL": "1"
+  },
+  "onCreateCommand": "curl -sSf https://get-ghcup.haskell.org | sh",
   "remoteUser": "vscode"
 }
```

環境変数 `BOOTSTRAP_HASKELL_NONINTERACTIVE` を `"1"` に設定することで，GHCup のインストールスクリプトを非対話的に実行します．また，`BOOTSTRAP_HASKELL_MINIMAL` を `"1"` に設定することで，GHCup だけをインストールし，GHC などのツールチェーンを追加でインストールしないようにします．

### 3.4. HLS のインストール

次に HLS をインストールする設定を追加します．GHCup とその管理下のツールは `~/.ghcup/bin` にインストールされるため，`ghcup` コマンドを実行できるよう PATH を追加します．`onCreateCommand` で `ghcup install hls` を実行し HLS をインストールします．

```diff jsonc:.devcontainer/devcontainer.json
 {
   "name": "Haskell",
   "image": "haskell:latest",
   "features": {
     "ghcr.io/devcontainers/features/common-utils:2": {}
   },
   "containerEnv": {
     "BOOTSTRAP_HASKELL_NONINTERACTIVE": "1",
     "BOOTSTRAP_HASKELL_MINIMAL": "1"
   },
+  "remoteEnv": {
+    "PATH": "/home/vscode/.ghcup/bin:${containerEnv:PATH}"
+  },
-  "onCreateCommand": "curl -sSf https://get-ghcup.haskell.org | sh",
+  "onCreateCommand": "curl -sSf https://get-ghcup.haskell.org | sh && ghcup install hls --set",
   "remoteUser": "vscode"
 }
```

`--set` オプションを付けることで，インストールした HLS をアクティブなバージョンとして設定します．

### 3.5. 拡張機能の設定

最後に Haskell の VS Code 拡張機能を設定します．

```diff jsonc:.devcontainer/devcontainer.json
 {
   "name": "Haskell",
   "image": "haskell:latest",
   "features": {
     "ghcr.io/devcontainers/features/common-utils:2": {}
   },
   "containerEnv": {
     "BOOTSTRAP_HASKELL_NONINTERACTIVE": "1",
     "BOOTSTRAP_HASKELL_MINIMAL": "1"
   },
   "remoteEnv": {
     "PATH": "/home/vscode/.ghcup/bin:${containerEnv:PATH}"
   },
   "onCreateCommand": "curl -sSf https://get-ghcup.haskell.org | sh && ghcup install hls --set",
+  "customizations": {
+    "vscode": {
+      "extensions": [
+        "haskell.haskell",
+        "haskell.language-haskell"
+      ]
+    }
+  },
   "remoteUser": "vscode"
 }
```

`haskell.haskell` は Haskell の公式拡張機能です．`haskell.language-haskell` は Haskell のシンタックスハイライトを提供する拡張機能です．以前は `justusadam.language-haskell` として公開されていましたが，現在は Haskell の公式拡張機能になっています．

完成した `devcontainer.json` は次のようになります．

```jsonc:.devcontainer/devcontainer.json
{
  "name": "Haskell",
  "image": "haskell:latest",
  "features": {
    "ghcr.io/devcontainers/features/common-utils:2": {}
  },
  "containerEnv": {
    "BOOTSTRAP_HASKELL_NONINTERACTIVE": "1",
    "BOOTSTRAP_HASKELL_MINIMAL": "1"
  },
  "remoteEnv": {
    "PATH": "/home/vscode/.ghcup/bin:${containerEnv:PATH}"
  },
  "onCreateCommand": "curl -sSf https://get-ghcup.haskell.org | sh && ghcup install hls --set",
  "customizations": {
    "vscode": {
      "extensions": [
        "haskell.haskell",
        "haskell.language-haskell"
      ]
    }
  },
  "remoteUser": "vscode"
}
```

## 4. 考察

通常の Dev Container では，ベースイメージに言語処理系を用意し，エディタ連携は VS Code 拡張機能に任せる，という役割分担ができます．Language Server そのものは Haskell に限った仕組みではなく，Rust の `rust-analyzer` や Go の `gopls` なども同じ位置づけです．

一方で Haskell では，GHC と HLS の対応関係が強いため，拡張機能を入れるだけでは完結せず，コンテナ内の実行環境に合わせて HLS も用意する必要があります．しかし HLS は IDE やエディタと連携するためのツールであり，言語処理系そのものではありません．そのため，Haskell の公式イメージに HLS が含まれていないこと自体は妥当だと考えられます．

今回の構成では，このギャップを Dev Container の `onCreateCommand` で埋めています．Haskell 本体は公式イメージに任せ，拡張機能との接続に必要な HLS だけをコンテナ作成時に追加するため，技術的にも自然な構成だと思います．この点は，HLS を Haskell のコアとなる公式イメージに含めるよりも，別のイメージや Dev Container 側で扱うほうが自然ではないか，という `haskell/docker-haskell` での議論とも整合しています．

## 5. おわりに

Dev Container による Haskell の環境構築では，現在でも公式テンプレートがない，公式イメージには HLS が含まれていない，Haskell Feature はコミュニティ版である，といった悩ましさがあります．

一方で，Haskell 公式イメージをベースにしつつ，HLS を GHCup 経由で `onCreateCommand` から追加すれば，Dockerfile を書かずに VS Code の Haskell 拡張機能まで動作する環境を構築できます．言語処理系は公式イメージに任せ，不足している開発支援ツールだけを追加するため，構成の責任範囲も比較的明確です．
